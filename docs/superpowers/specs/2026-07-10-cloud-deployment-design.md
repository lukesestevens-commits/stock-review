# 复盘网站云端部署与同步设计

## 目标

把现有复盘网站部署为手机可直接访问的 HTTPS 网站，并保持电脑端同花顺捕获流程不变。电脑端复盘助手继续抓取当天资金、持仓和交易记录，随后自动上传云端；手机打开网站后使用访问码读取并自动填写同一份当天数据。

## 部署方案

使用 OpenAI Sites 托管网站和 API，使用 Sites 提供的 D1 数据库保存最新同步快照。站点公开可访问，所有包含账户数据的云同步 API 必须验证独立访问码。首版使用 Sites 分配的域名，不绑定自定义域名。

现有单文件复盘页面继续作为主要界面。部署适配层负责静态页面托管、云同步 API、市场快照 API 和 D1 访问，不改变本地同花顺扩展的捕获职责。

## 数据流

1. 电脑端同花顺扩展把当天接口数据发送到本地复盘助手。
2. 本地复盘助手按交易日合并数据并保留本地副本。
3. 配置了云端地址和访问码后，本地助手把合并后的快照上传到 `POST /api/sync/tzzb`。
4. 云端验证访问码，只接收目标交易日的数据，按当天记录进行合并并写入 D1。
5. 手机端页面请求 `GET /api/sync/health` 和 `GET /api/sync/latest`，验证访问码后读取最新快照并自动填写复盘表单。

云端首版只保存最新交易日快照。收到新的交易日数据时覆盖旧交易日；同一天的多次上传继续合并，避免分批接口响应导致字段缺失。

## 云端数据模型

D1 使用单表保存最新快照：

- `id`: 固定主键，用于原子更新最新记录。
- `target_date`: 快照所属交易日。
- `received_at`: 云端最后接收时间。
- `payload_json`: 完整合并后的捕获数据。

结构化查询不是首版需求，因此完整快照保存在 JSON 文本字段中。后续需要多日历史时再扩展为按交易日分行，不在本次范围内。

## 接口与访问控制

- `POST /api/sync/tzzb`: 电脑端上传当天快照。
- `GET /api/sync/latest`: 手机端读取可直接填写的复盘数据。
- `GET /api/sync/health`: 返回同步日期、接收时间、记录数量和数据完整度。
- `GET /api/market-snapshot`: 保持页面现有市场数据自动填写能力。

访问码由 Sites 环境变量 `TZZB_SYNC_ACCESS_KEY` 提供。上传优先使用 `X-TZZB-Sync-Key` 请求头；手机读取使用同一访问码。接口不在日志或响应中回显访问码。未配置访问码时接口拒绝服务，访问码错误时返回 401。

网页本身不包含账户数据。复盘草稿仍保存在手机浏览器本地；云端只保存同花顺同步快照。访问码首次输入后保存在当前设备浏览器中，用户无需每次重输。

## 手机端行为

线上环境默认使用当前网站地址作为云同步地址，用户只需要选择“云端同步”并输入访问码。桌面本地环境仍默认使用本地助手，避免影响现有使用方式。

页面保持现有响应式布局和自动填写流程。云端暂无数据、数据不是当天、数据完整度不足或访问码错误时，页面显示明确状态，不覆盖用户已经填写的内容。

## 电脑端配置

部署完成后，在本机 `云同步配置.env` 写入最终站点地址和生成的访问码。启动器继续读取该文件，复盘助手每次收到同花顺数据后自动上传。云端失败不影响本地保存，并在助手响应和日志中保留可诊断的同步结果。

## 失败处理

- 云端上传失败：本地捕获成功，记录云同步失败信息，下一次捕获时再次上传。
- 云端没有快照：返回 404 和可读提示。
- 非当天数据：拒绝用于自动填写，避免旧数据覆盖当天复盘。
- D1 暂时不可用：返回 503，不伪装成同步成功。
- 市场数据源失败：保持现有手动填写能力，不影响持仓和交易同步。

## 验证

实施完成后验证以下路径：

1. 云端构建成功，手机尺寸页面可加载且主要控件不溢出。
2. 无访问码和错误访问码无法上传或读取账户数据。
3. 正确访问码可以上传、查询健康状态并读取复盘映射结果。
4. 同一天多次上传正确合并，新交易日会替换旧快照。
5. 本地助手配置后能把真实格式的捕获数据上传到线上地址。
6. 手机端选择云端同步后能自动填写资金、持仓和交易记录。
7. 本地模式和现有测试继续通过。

## 验收标准

- 获得一个手机可访问的公开 HTTPS 地址。
- 账户数据只能通过正确访问码读取。
- 电脑端启动现有助手后，无需手动导出文件即可把当天数据同步到云端。
- 手机刷新页面后仍能读取云端最新数据并完成自动填写。
- 最终站点地址和访问码已经写入本机配置，并完成一次端到端在线验证。

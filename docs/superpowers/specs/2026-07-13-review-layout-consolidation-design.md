# 复盘模块与成交列精简设计

## 背景

页面当前同时存在“持仓复盘与明日预案”和独立的“明日操作计划”。持仓表已经逐只记录是否核心、持仓逻辑、明日处理和触发条件，独立模块中的“主线核心仓怎么处理”和“随手单 / 低质量仓怎么处理”与这些字段重复。“给明天的自己一句话”不属于必要的操作信息，也可以删除。

成交记录目前把“买/卖”“金额”“明细”拆成三个表格列。“明细”中又包含价格和数量，导致表格横向字段过多，桌面端信息密度低，窄屏需要更长距离横向滚动。

## 目标

- 将所有明日操作信息集中到“持仓复盘与明日预案”。
- 删除重复或非必要字段，缩短页面主流程。
- 将成交记录的三个相关列合并为一个紧凑的“交易”列。
- 保持同花顺同步、交易评分、风险计算、本地保存、历史记录和文字导出正常。
- 缩短成交表的最小宽度，改善桌面端和移动端浏览效率。

## 非目标

- 不改变同花顺成交与持仓数据的抓取、映射或云端同步方式。
- 不改变交易评分维度和计算方法。
- 不删除成交价格、数量或金额数据。
- 不改变市场环境、今日反思和历史追踪模块的业务含义。

## 方案比较

### 合并列并折叠次要字段（采用）

“交易”列常显买卖方向与成交价格，数量和金额放入同一折叠区。价格是判断交易位置最常查看的信息，因此保持可见；数量和金额仍可编辑、自动计算并参与风险检查，但默认不占用额外表格列。

### 所有交易字段在一个单元格内常显

信息无需展开即可查看，但每行高度会明显增加，成交较多时浏览效率下降。

### 删除数量或金额

最精简，但会破坏成交复核、金额自动计算和单笔仓位风险判断，不采用。

## 页面结构

### 持仓复盘与明日预案

保留现有持仓表字段：

- 股票 / ETF。
- 市值 / 仓位。
- 是否核心。
- 持仓逻辑是否成立。
- 明日处理。
- 触发条件。

在持仓表下方增加两项全局计划：

- “明日允许新开的仓位 / 方向”。
- “明日禁令”。

删除以下内容：

- 独立“明日操作计划”模块。
- “主线核心仓怎么处理”。
- “随手单 / 低质量仓怎么处理”。
- “给明天的自己一句话”。
- “生成明日计划”按钮及其自动汇总函数。

持仓表本身已逐只表达核心与非核心仓的处理方式，因此不再生成重复的概括文本。

### 成交记录

表头中的“买/卖”“金额”“明细”合并为“交易”。每行对应单元格包含：

1. 常显区域：买入 / 卖出选择器、成交价格。
2. 折叠区域：成交数量、成交金额。

价格或数量变化时继续自动计算金额；用户也可以展开后直接修正金额。底层 CSS 类名和交易数据结构保持不变，评分、风险分析、导入和导出继续读取同一字段。

## 数据兼容

新保存记录的 `plan` 仅包含 `newPlan` 和 `banRule`。读取旧记录时继续读取这两个字段；旧记录中的 `corePlan`、`noisePlan` 和 `selfTalk` 被安全忽略，不影响其他字段加载。

交易对象仍保留 `side`、`price`、`qty` 和 `amount`，只改变页面呈现方式。现有同花顺同步与 OCR 导入不需要数据迁移。

文字导出将“明日允许新开的仓位 / 方向”和“明日禁令”并入“持仓复盘与明日预案”，不再生成独立“明日操作计划”章节。

## 响应式与可用性

- 成交表减少两个表格列，并相应降低最小宽度。
- “交易”单元格使用紧凑网格，买卖方向和价格在桌面与窄屏均保持清晰。
- 折叠控件继续支持鼠标、键盘和触摸操作。
- 持仓表下方的两项全局计划在宽屏并排，在窄屏自动单列。

## 测试策略

- 页面结构测试：独立“明日操作计划”、三个被删除字段和生成按钮不再出现。
- 持仓测试：新开仓计划与明日禁令位于持仓模块且可保存、恢复和导出。
- 成交表测试：只出现一个“交易”表头，买卖方向和价格常显，数量与金额位于同一折叠区。
- 数据测试：交易字段、金额自动计算、评分和风险判断保持原有结果。
- 响应式测试：成交表最小宽度缩短，现有移动端安全区与横向滚动规则继续通过。
- 完整回归：运行全部页面、同步、云端和构建测试。

## 验收标准

- 页面不再显示独立“明日操作计划”模块。
- 持仓模块下只保留新开仓方向 / 仓位和明日禁令两项全局计划。
- 成交表以一个“交易”列替代原来的买卖、金额和明细三列。
- 买卖方向与价格默认可见，数量与金额默认折叠且可编辑。
- 同花顺同步、保存恢复、历史记录、文字导出、评分和风险计算不受影响。
- 桌面端和移动端布局测试及完整回归全部通过。
